## 1. 设计结论

本项目在不更换现有“两室两厅一卫”沙盘、不改变既有传感器种类和固定 GPIO 接线的前提下，建设一套“感知—判断—执行—显示—证据”的家庭生活规则实验室。

作品名称固定为：

> **N16R8 节气智居——一屋一策家庭生活规则实验室**

核心创新不是增加硬件，而是让二十四节气真正影响窗帘和补光策略，并建立断网仍有效的本地安全优先级。规定任务、创新功能和答辩证据必须来自同一套真实主板状态，网页不得用模拟数据冒充真板在线。

### 4.1 电气安全合同

- 旋钮使用 `3.3V` 供电，GPIO17 输入必须保持在 `0~3.3V`。
- MQ2 的 AO 在进入 GPIO2 前必须经过限压或分压，最高不得超过 `3.3V`。
- 风扇、舵机或继电器使用外部电源时，外部电源必须与 N16R8 共地。
- 继电器模块必须支持 `3.3V` 控制信号，触发极性通过空载测试确定。
- 继电器触点只连接低压 LED 灯或模型负载，禁止连接 `220V` 市电。
- 上电时风扇关闭、继电器关闭、蜂鸣器静音、RGB 进入安全状态；初始化完成后才进入 `Auto`。

## 5. 总体架构

系统分为三个互不混淆的规则层：

1. **规定任务层**：严格完成 T、Q、N、H 显示、按键/语音切换、Auto/Sleep、旋钮阈值五项任务。
2. **节气策略层**：在无安全告警且处于 Auto 时，根据二十四节气调整窗帘和补光策略，并给出温度建议。
3. **安全底线层**：由主板本地判断 MQ2、火焰、水浸和布防状态下的 PIR；断网时仍能执行报警联动。

数据闭环为：

```text
传感器/旋钮/按键
      ↓
N16R8 本地采样、滤波、规则和优先级
      ↓
风扇/舵机/继电器/RGB/蜂鸣器/OLED
      ↓
hello / telemetry / event / ack
      ↓
电脑评分控制台与语音入口
      ↓
白名单 command 返回 N16R8，再由新 telemetry 证明结果
```

## 6. 五项规定功能逐项映射（50 分）

### 6.1 功能一：OLED 显示 T、Q、N、H（10 分）

普通监控页固定采用四行：

```text
Auto
T:27.4 c RH:61%
Q:0386 ppm
N:63 H:1
```

- `T`：DHT11 温度。
- `Q`：MQ2 等效估算值。OLED 按评分格式显示 `ppm`，网页和答辩必须明确它不是标准 AQI，也不是实验室标定的通用空气质量值。
- 等效值采用透明、可复现的演示映射：`Q = round(clamp(mq2RawFiltered, 0, 4095) × 1000 ÷ 4095)`，范围为 `0~1000` 等效 ppm，告警阈值初始固定为 `600`。该线性映射只服务本沙盘的相对变化展示。
- `N`：声音相对强度 `0~100`。
- `H`：PIR 活动状态，`1` 表示检测到活动，`0` 表示未检测到活动。
- `RH` 是额外信息，不替代 T/Q/N/H 四项规定字段。

DHT11 独立每 `2000ms` 读取一次；其他快速传感器可每 `200ms` 采样并滤波；遥测建议每 `1000ms` 输出一次。

### 6.2 功能二：按键、语音和温控风扇（10 分）

- A 键每次有效单击切换一次：`Auto → Sleep → Auto`。
- 按键必须消抖；长按不重复触发模式切换。
- 语音只接受白名单意图：“打开自动模式”“切换到自动模式”“打开睡眠模式”“切换到睡眠模式”。语音最终转换为与网页相同的 `command`，不能直接修改网页假状态。
- Auto 温控采用 `±0.5℃` 迟滞：
  - `T >= YZ + 0.5℃`：风扇 `100%`。
  - `T <= YZ - 0.5℃`：风扇 `0%`。
  - 中间区间：保持上一次 Auto 风扇状态，防止频繁启停。
- 风扇高、低、关闭三种状态必须肉眼可区分。

### 6.3 功能三：电脑端显示 T、Q、N、H（10 分）

电脑端设置独立“评分数据台”，字段固定为：

```text
T:27.4c
Q:386 ppm
N:63
H:1
```

页面只显示最新且未过期的真板遥测。超过 `3500ms` 没有新 `hello` 或 `telemetry` 时，显示“开发板离线/等待实时数据”，不得继续把最后数值包装为在线实时值。

### 6.4 功能四：Auto 与 Sleep 模式（10 分）

| 项目 | Auto | Sleep |
| --- | --- | --- |
| OLED 首行 | `Auto` | `Sleep` |
| 风扇 | 根据温度阈值自动 `0%/100%` | 固定低速，初始目标 `35%` |
| 黄色灯 | RGB 黄色常亮 | RGB 关闭 |
| 节气策略 | 可控制窗帘和补光 | 暂停普通节气动作 |
| 安全规则 | 始终有效 | 始终有效并可覆盖低速风扇 |

Sleep 低速需要在真板上从 `30%~45%` 范围校准，最终值必须做到稳定转动且与 Auto 高速明显不同；规格初始值固定为 `35%`，只有真板校准记录可以修改它。

### 6.5 功能五：旋钮调整温度阈值（10 分）

- GPIO17 原始值记为 `XN`，有效范围 `0~4095`。
- 温度阈值记为 `YZ`，映射范围固定为 `18~35℃`。
- 对 ADC 原始值使用中值或滑动平均滤波，并设置端点死区，防止显示抖动。
- 阈值页至少显示：

```text
XN:2048
YZ:27
```

- 演示时转动旋钮，使 `YZ` 穿越当前 `T`，在不等待环境温度变化的情况下证明风扇判断规则确实改变。
- GPIO17 连续异常时显示 `XN:--`；若已经有有效阈值则保持最后有效值，若开机后从未成功采样则使用安全默认值 `27℃`。

## 8. 创新功能一：二十四节气舒适策略

节气不是图片或口号，而是 Auto 模式下的可执行参数表。每个节气包含三个字段：温度建议值、窗帘偏置和补光阈值。温度建议只用于解释和提示，GPIO17 旋钮产生的 `YZ` 始终是最终 Auto 温控阈值，节气不得悄悄覆盖用户设置。

光照值按 `0=暗、100=亮` 归一化。Auto 且有人活动时，光照低于节气补光阈值才允许打开低压学习灯；Sleep 或安全告警时继电器关闭。窗帘偏置以逻辑开合百分比表达：`20%` 偏向引光、`50%` 平衡、`80%` 偏向遮阳，最终舵机角度通过实物端点校准并限位。

## 9. 创新功能二：断网仍工作的家庭安全底线

安全判断完全在 N16R8 本地运行，不依赖网页、语音、WSS 或 MQTT。

| 风险来源 | 本地动作 | 必须上报的告警 |
| --- | --- | --- |
| MQ2 超阈值 | 风扇 100%、RGB 红、蜂鸣器报警、继电器关闭 | `mq2` |
| 火焰触发 | 风扇 100%、RGB 红、蜂鸣器报警、继电器关闭 | `flame` |
| 水浸触发 | 风扇 100%、RGB 红、蜂鸣器报警、继电器关闭 | `water` |
| 布防开启且 PIR 活动 | RGB 红、蜂鸣器报警；不虚构风扇联动 | `intrusion` |

正式演示只用水浸传感器：先进入 Sleep，给 GPIO8 水浸探头滴少量水，展示风扇 `35%→100%`、RGB `关闭→红色`、蜂鸣器报警、OLED/网页告警同步。演示不点火、不释放燃气。

风险解除后，系统必须在连续安全样本满足消抖条件后清除告警，再根据当前 `Auto/Sleep`、当前节气和当前传感器重新计算动作，禁止恢复告警前保存的旧 GPIO 快照。解除条件固定为：水浸连续干燥 `3000ms`；火焰连续未触发 `3000ms`；MQ2 等效值连续低于 `550` 达 `3000ms`；入侵告警在 PIR 连续无活动 `5000ms` 或用户关闭布防后解除。

## 10. 完整优先级

从高到低固定为：

| 优先级 | 名称 | 规则 |
| --- | --- | --- |
| P0 | 上电安全 | 执行器先进入安全初值，传感器和协议完成初始化 |
| P1 | 安全告警 | MQ2、火焰、水浸和布防入侵覆盖其他所有动作 |
| P2 | 规定模式 | Auto/Sleep 的风扇、OLED 和黄色灯规则 |
| P3 | 节气优化 | 仅在 Auto 且无 P1 时控制窗帘和补光 |
| P4 | 普通手动/调试 | 只允许白名单动作，不得压过 P1 |

安全期间，直接改变风扇、RGB、继电器或蜂鸣器的手动命令返回 `ack.ok=false` 和 `safety_lock`。`Auto/Sleep` 模式命令可以被接受并更新目标模式，但返回的成功 `ack` 必须带 `"deferredBy":"safety"`；P1 动作保持不变，告警解除后才按新目标模式重新计算。

## 11. 数据协议与证据闭环

所有串口协议均为一行一个 JSON 对象，调试文字不得混入协议行。

### 11.1 hello

```json
{"type":"hello","protocolVersion":1,"project":"smartlife-junior","profileId":"smartlife-junior-solar-home-v1","board":"n16r8_esp32s3","baud":115200,"rfid":false}
```

### 11.2 telemetry

```json
{"type":"telemetry","protocolVersion":1,"seq":42,"uptimeMs":185000,"mode":"Auto","solarTerm":"小暑","sensors":{"temperatureC":27.4,"humidityRh":61,"airQualityEqPpm":386,"soundRelative":63,"presence":true,"lightRelative":44,"water":false,"flame":false,"knobRaw":2048},"thresholds":{"temperatureC":27,"mq2EqPpm":600},"actuators":{"fanPercent":100,"curtainClosePercent":80,"relay":false,"buzzer":false,"rgb":"yellow"},"display":{"page":"score","lines":["Auto","T:27.4 c RH:61%","Q:0386 ppm","N:63 H:1"]},"alerts":[],"health":{"dht":"ok","mq2":"ready","knob":"ok","oled":"ready","buzzerEnabled":true},"lastAppliedCommandId":"cmd-001"}
```

`airQualityEqPpm` 的字段名强制保留 `Eq`，网页必须显示“MQ2 等效估算”，不得在内部字段中伪装成标准空气质量值。

### 11.3 command

```json
{"type":"command","id":"cmd-001","origin":"dashboard","mode":"Sleep"}
{"type":"command","id":"cmd-002","origin":"voice","set":{"solarTerm":"大寒"}}
{"type":"command","id":"cmd-003","origin":"dashboard","set":{"guardArmed":true}}
```

- `id` 必须唯一。
- `origin` 白名单为 `dashboard`、`voice`、`test`。
- 模式只接受 `Auto`、`Sleep`。
- 节气只接受二十四节气表中的准确名称。
- 重复命令 ID 返回第一次的相同 `ack`，不得重复驱动执行器。

### 11.4 ack

```json
{"type":"ack","id":"cmd-001","ok":true,"applied":{"mode":"Sleep"}}
{"type":"ack","id":"cmd-005","ok":true,"applied":{"mode":"Auto"},"deferredBy":"safety"}
{"type":"ack","id":"cmd-004","ok":false,"error":"safety_lock","message":"安全告警期间不能覆盖执行器"}
```

`ack` 只证明主板接受或拒绝命令。完整证据等级为：

1. 收到 `ack`：主板接受命令。
2. `ack` 后收到带相同 `lastAppliedCommandId` 的新遥测：主板已经重新计算状态。
3. 新遥测中的执行器字段符合预期：软件闭环成立。
4. 风扇、灯、舵机、RGB 或蜂鸣器的实物动作可见：实物闭环成立。

继电器遥测只证明 GPIO 输出目标状态；没有触点反馈传感器时，不得声称低压负载一定已经通断。

### 11.5 本地事件

本地按键和旋钮变化使用 `event`，不能伪造为远程命令的 `ack`：

```json
{"type":"event","event":"button","key":"A","action":"toggle_mode","mode":"Sleep"}
{"type":"event","event":"threshold_changed","source":"knob","knobRaw":2300,"temperatureThresholdC":28}
{"type":"event","event":"alert_changed","active":["water"]}
```

## 12. 故障处理与降级

| 故障 | 主板行为 | OLED/网页行为 |
| --- | --- | --- |
| DHT11 单次失败 | 保留最后有效值 | 标记瞬时异常，不立即清空 |
| DHT11 超过 `6000ms` 无有效值 | 温湿度置 `null`，停止依赖温度的新 Auto 判断并保持当前安全输出 | 显示 `T:--` 和 `dht:stale` |
| MQ2 预热 | 不产生正常 Q 值，也不产生预热假告警 | `Q:---- ppm`、`mq2:warming` |
| GPIO17 异常 | 保留最后有效阈值；无历史值时使用 `27℃` | `XN:--`，YZ 显示当前有效阈值 |
| OLED 初始化失败 | 其他本地规则继续运行 | `health.oled=error`，网页明确提示 |
| 网页断开 | 本地模式、旋钮和安全规则继续运行 | 超过 `3500ms` 显示离线 |
| 未知命令/越界值 | 不改变状态 | `ack.ok=false`，返回 `unsupported_command` 或 `out_of_range` |
| 安全期间手动执行器命令 | 保持 P1 动作 | `ack.ok=false`、`safety_lock` |
| 安全期间 Auto/Sleep 命令 | 更新目标模式但暂不改变 P1 动作 | `ack.ok=true`、`deferredBy:safety` |

MQ2 固件最短预热门限固定为 `60000ms`；比赛当天应提前至少 10 分钟通电稳定。这个预热安排只降低漂移，不构成标准 ppm 校准证明。

## 17. 关键风险与通过条件

### 17.1 MQ2 的 ppm 证据风险

评分格式要求 `Q:数值 ppm`，但现有 MQ2 只能在未做气体种类、负载电阻和标准气体校准时提供相对变化。方案采用 `airQualityEqPpm` 满足显示格式，同时在网页、讲稿和答辩中明确“MQ2 等效估算”。如果评委要求可溯源的标准 ppm，本硬件条件不能诚实证明，必须以现场规则解释或增加合规标定证据，不能改文案掩盖。

### 17.2 GPIO17 与 Wi-Fi 稳定性

GPIO17 属于 ESP32-S3 ADC2 通道。必须在最终连接路线、Web Serial/Wi-Fi 功能实际开启的组合状态下复测端点、噪声和丢读；只在离线程序中读数成功不算通过。

### 17.3 继电器触发极性

不同继电器模块可能高电平或低电平触发。接入低压负载前先空载上电，确认启动不吸合，并把逻辑状态、指示灯、触点动作和遥测分别记录。

### 17.4 实物证据边界

- 页面按钮变化不等于主板动作。
- `ack` 不等于执行器动作。
- GPIO 目标值不等于继电器触点反馈。
- 通过软件测试不等于真板完成。
- “初中”或“初中2组”的已有实现不等于“初中3”已经实现。
