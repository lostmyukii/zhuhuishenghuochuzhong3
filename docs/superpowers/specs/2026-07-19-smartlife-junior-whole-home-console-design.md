# 节气智居·全屋运行台——任务八设计规格

> 日期：2026-07-19
>
> 状态：A 方案已由用户确认，待书面规格复核后实施
>
> 对应主规格：`2026-07-19-smartlife-junior-solar-home-design.md`

## 1. 页面定位

页面名称固定为：

> **节气智居 · 全屋运行台**

页面面向操作沙盘的学生、老师和家庭观察者，唯一任务是把“两室两厅一卫”的实时状态、家庭模式、安全告警和设备联动集中显示并允许安全控制。

页面不是竞赛后台。可见界面禁止出现“评分、得分、满分、评委、评分数据台、评分操作”等表达；评分细则与验收映射只保留在项目文档中，不进入家庭运行界面。

## 2. 已选方案与取舍

已选择 A“全屋运行台”：以住宅剖面图为主视觉，把数据与房间职责对应起来。

- 不采用纯设备数据台：它难以体现现有沙盘的空间关系。
- 不采用单线场景演示页：它适合讲故事，但不适合同时观察安全状态和全屋设备。
- 不制作欢迎页或宣传首页：打开页面后直接进入实时运行界面。

## 3. 信息架构

桌面端由上到下分为五层：

1. **连接状态栏**：连接串口、断开、串口状态、开发板状态、数据来源、最后帧时间。
2. **全屋剖面图**：主卧、次卧/书房、客厅、餐厅、卫生间，以及厨房、入户、阳台附区。
3. **实时数据与家庭模式**：温度、MQ2 等效估算、声音、人员活动、Auto/Sleep、GPIO17 阈值。
4. **节气与安全**：二十四节气、窗帘/补光策略、安全原因、来源 GPIO 和实际目标动作。
5. **运行记录**：command、ack、event、新 telemetry 和命令完成状态。

桌面布局：

```text
┌──────────────── 连接状态与全屋模式 ────────────────┐
├──────────────────────┬────────────────────────────┤
│                      │ 安全通知 / 当前节气        │
│ 两室两厅一卫剖面图   ├────────────────────────────┤
│ 房间热点与设备状态   │ T / Q / N / H / XN / YZ   │
│                      │ Auto / Sleep / 布防        │
├──────────────────────┴────────────────────────────┤
│ 设备目标状态                  │ 运行记录            │
└───────────────────────────────────────────────────┘
```

`390px` 移动端改为单列：连接状态 → 安全通知 → 房间图 → 实时数据 → 模式/节气 → 设备状态 → 运行记录，不允许横向滚动。

## 4. 房间与硬件映射

| 空间 | 页面重点 | 数据或设备 |
| --- | --- | --- |
| 主卧 | 睡眠与节气窗帘 | 舵机 GPIO9、窗帘目标开合度 |
| 次卧/书房 | 学习环境与低压灯 | 声音 GPIO4、继电器 GPIO12 |
| 客厅 | 全屋主控与舒适度 | DHT11 GPIO14、风扇 GPIO11、RGB GPIO46、A 键 GPIO10、旋钮 GPIO17、OLED GPIO41/42 |
| 餐厅 | 节气生活策略 | 当前节气、温度建议、全屋风扇策略 |
| 卫生间 | 漏水安全 | 水浸 GPIO8 |
| 厨房附区 | 烟雾/可燃气体与火焰安全 | MQ2 GPIO2、火焰 GPIO45 |
| 入户附区 | 布防活动 | PIR GPIO5 |
| 阳台附区 | 自然光输入 | 光照 GPIO1 |

房间热点只呈现当前协议中存在的数据和目标状态，不虚构每个房间都有独立传感器。

## 5. 视觉系统

视觉方向为“住宅剖面蓝图 + 节气签”，不采用通用深色监控大屏。

| Token | 颜色 | 用途 |
| --- | --- | --- |
| 月白 | `#F2F4EF` | 页面底色 |
| 墨青 | `#17363B` | 主文字、建筑结构线 |
| 青瓷 | `#4F7C72` | 正常运行、模式状态 |
| 日光 | `#E5A43A` | Auto 黄色灯、节气重点 |
| 连线蓝 | `#2F6F91` | 串口连接、参与中的传感器 |
| 警戒红 | `#C64738` | P1 安全告警 |

字体均使用本机字体，不依赖外部 CDN：标题/节气签使用 `STKaiti, KaiTi, serif`，正文使用 `PingFang SC, Microsoft YaHei, sans-serif`，协议和数值使用 `SFMono-Regular, Consolas, monospace`。

页面唯一的强识别元素是全屋剖面图中的“节气气流线”：正常时以低对比青瓷色连接当前参与房间，模式或节气真正改变时动画一次；安全告警时停止普通动画并把对应风险路径改为红色。`prefers-reduced-motion` 下取消位移动画，保留颜色、图标和文字状态。

## 6. 数据真实性与连接状态

连接状态分开显示，不能互相代替：

- **串口未连接**：浏览器没有打开串口。
- **串口已连接，等待开发板**：串口打开，但尚未收到匹配身份的 `hello/telemetry`。
- **真板在线**：收到目标 `profileId=smartlife-junior-solar-home-v1` 的 hello，并持续收到新 telemetry。
- **模拟主板**：帧带 `source=mock`，页面顶部和房间图持续显示“模拟数据，不代表实物动作”。
- **开发板离线**：距最新 hello/telemetry 超过 `3500ms`。

离线后，实时卡片显示 `--` 和“等待实时数据”；最后一帧可保留在运行记录中，但不能继续显示为当前正常或当前告警结论。坏 JSON 只写入解析错误记录，不改变最后有效状态。

页面必须先收到 `profileId=smartlife-junior-solar-home-v1` 的 hello 才锁定本次会话；随后允许接收同一串口中不重复携带 `profileId` 的 telemetry/event/ack。其他身份显示“不是本项目开发板”，不更新房间、传感器或执行器状态。

浏览器没有 `navigator.serial` 或当前页面不是可使用 Web Serial 的安全上下文时，直接说明需要本机 Chrome/Edge 和 localhost/HTTPS，不显示虚假的连接按钮成功状态。

## 7. 实时数据与文案

实时区保留协议简写，主标签使用家庭语言：

| 简写 | 主标签 | 展示规则 |
| --- | --- | --- |
| T | 室内温度 | `27.4℃`；无有效值显示 `--` |
| Q | 空气安全参考 | `386 ppm`，旁边固定显示“MQ2 等效估算，非标准检测仪读数” |
| N | 环境声音 | `63`，明确为相对强度，不写 dB |
| H | 人员活动 | `有人活动/未检测到活动`，明确 PIR 不识别人 |
| XN | 旋钮原始值 | `0~4095` |
| YZ | 温控阈值 | `18~35℃` |

继电器只写“低压灯 GPIO 目标：开/关”，不得写成“触点已闭合/已断开”。执行器卡片展示的是开发板上报目标状态；真机动作仍需现场观察。

## 8. 模式、节气与命令闭环

主页提供 Auto、Sleep、布防、蜂鸣器安全静音和二十四节气选择。语音入口留到任务九，本任务不创建假语音按钮。

每次网页命令使用唯一 ID，并按以下状态显示：

1. `正在发送`：命令已写入串口。
2. `等待开发板接受`：尚未收到同 ID ACK。
3. `已接受，等待状态更新`：`ack.ok=true`，但尚无更新后的 telemetry。
4. `状态已更新`：新 telemetry 的 `lastAppliedCommandId` 等于该命令 ID。
5. `开发板拒绝`：显示 `ack.error` 和中文原因。
6. `等待超时`：明确指出没有完成闭环，不修改页面假状态。

安全告警期间，执行器操作不得覆盖 P1；模式命令出现 `deferredBy=safety` 时显示“模式已记录，安全状态解除后生效”。

## 9. 安全通知

红色安全状态只来自 `telemetry.alerts`，不能由 PIR、水浸或 MQ2 原始值在浏览器自行推断。

| 告警 | 房间/来源 | 页面动作摘要 |
| --- | --- | --- |
| `mq2` | 厨房附区 GPIO2 | 风扇 100%、RGB 红、继电器关闭；按 telemetry 判断蜂鸣器是否正在报警 |
| `flame` | 厨房附区 GPIO45 | 风扇 100%、RGB 红、继电器关闭；按 telemetry 判断蜂鸣器是否正在报警 |
| `water` | 卫生间 GPIO8 | 风扇 100%、RGB 红、继电器关闭；按 telemetry 判断蜂鸣器是否正在报警 |
| `intrusion` | 入户附区 GPIO5 | RGB 红；按 telemetry 判断蜂鸣器是否正在报警，不虚构风扇联动 |

未知告警显示为“设备上报异常：原始代码”，不能静默丢弃。安全通知使用 `role=status`、`aria-live=polite` 和 `aria-atomic=true`。

## 10. 文件边界

- `dashboard/protocol-core.js`：JSON 分类、项目身份、新鲜度、命令闭环状态；纯函数、无 DOM。
- `dashboard/alert-core.js`：告警去重、房间/GPIO/原因/动作摘要；纯函数、无 DOM。
- `dashboard/app.js`：Web Serial、状态机调度和 DOM 更新。
- `dashboard/index.html`：稳定语义结构和可访问标签。
- `dashboard/style.css`：住宅剖面视觉、响应式、焦点与 reduced-motion。
- `dashboard/tests/*.test.js`：Node 合同测试，不依赖真实串口。

本任务不建设公网、WSS、MQTT、Service Worker、语音或真板烧录。

## 11. 验收条件

1. hello/telemetry/event/ack 分类、身份锁定和坏 JSON 保护测试通过。
2. 页面源码和可见动态文案不存在竞赛评分化用语。
3. 3500ms 离线、模拟水印、真板等待状态均可测试。
4. 命令必须走 ACK 加新 telemetry 闭环；只收到 ACK 时不得显示已完成。
5. 告警完全以 `telemetry.alerts` 为准，未知代码可见。
6. 任务七模拟帧能驱动房间、传感器、执行器和安全通知。
7. 390px 无水平溢出，键盘焦点可见，颜色之外始终有文字状态。
8. 所有现有 Python/固件测试和 N16R8 编译继续通过。

通过本任务只能证明本地网页与模拟协议闭环，不能证明真实 GPIO 或执行器已经动作；在真板 hello/telemetry 出现前，页面不得显示“真板在线”。
