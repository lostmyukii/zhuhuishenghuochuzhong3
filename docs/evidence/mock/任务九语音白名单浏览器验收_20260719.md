# 任务九：语音白名单与安全控制浏览器验收

> 日期：2026-07-19
>
> 证据等级：浏览器与模拟协议证据；不是实际麦克风识别、真板串口或实物动作证据

## 1. 本次范围

- 在现有“节气智居 · 全屋运行台”中增加浏览器语音入口和独立文字测试入口。
- 语音识别文字必须再次经过本地纯白名单，不能直接生成任意 JSON 命令。
- 只允许以下四句固定口令：
  - `打开自动模式`
  - `切换到自动模式`
  - `打开睡眠模式`
  - `切换到睡眠模式`
- 通过白名单后只生成 `origin=voice` 的 `mode=Auto/Sleep` 命令；不能通过语音关闭安全判断或直接控制风扇、继电器、RGB、蜂鸣器。
- 命令继续使用任务八的唯一 ID、ACK、新 telemetry 和 `lastAppliedCommandId` 闭环。

## 2. 自动化测试

- `dashboard/tests/voice-intent-core.test.js`：`10/10` 通过。
  - 固定 Auto/Sleep 口令映射正确。
  - 空白、无关、含糊、否定、混合和危险话语均返回 `unknown`。
  - 只在句子中包含白名单子串不能通过。
  - 未通过白名单时不产生 `command`。
  - 合法命令固定为 `origin=voice`，ID 为空或超过 64 字符时拒绝。
  - 浏览器权限、无麦克风、无语音和网络失败均有明确中文说明。
- 协议测试 `9/9`、告警测试 `6/6`、页面合同和 11 帧模拟回放通过。
- Python 合同与模拟器测试 `27/27` 通过。
- 固件原生测试 `24/24` 通过。
- N16R8 目标编译通过：RAM `6.5%`，Flash `5.4%`。

## 3. Chromium 浏览器验收

本地入口：`http://127.0.0.1:8767/dashboard/`

1. 浏览器正确显示“语音白名单”、四句固定口令、麦克风入口、文字测试入口和 P1 安全说明。
2. 文字输入“关闭安全报警”后显示“未通过本地白名单”，模拟传输数组保持零发送。
3. 用模拟 `SpeechRecognition` 事件产生“打开睡眠模式”，页面生成且只生成：

   ```json
   {"type":"command","id":"voice-...","origin":"voice","mode":"Sleep"}
   ```

4. 普通模拟闭环依次显示“等待开发板接受”“开发板已接受，等待新遥测”“语音命令闭环完成”。
5. 模拟 `ack.ok=false` 时显示“开发板拒绝语音命令”，房间模式保持原来的 `Sleep`，没有假切换为 `Auto`。
6. 水浸 P1 期间发送 Sleep：
   - ACK 带 `deferredBy=safety` 后显示“模式已记录，P1 安全状态仍优先”。
   - 匹配新遥测后显示“模式目标已更新，P1 安全联动继续”。
   - 页面仍显示水浸告警、风扇 `100%`、继电器 GPIO 目标关、蜂鸣器目标报警和 RGB 红色，语音没有覆盖安全输出。
7. `390px` 视口实测：`innerWidth=390`、`documentElement.scrollWidth=390`、`body.scrollWidth=390`、语音面板宽度 `370px`。
8. Chromium 控制台：`0` 个错误、`0` 个警告。

## 5. 真实性边界

- Chromium 报告浏览器语音接口可用，但本次没有采集真人麦克风音频，也没有把浏览器在线语音服务结果当成现场语音证据。
- 浏览器中的麦克风事件链使用可重复的模拟 `SpeechRecognition` 结果验证；文字测试只验证本地解析器。
- 本次没有选择真实 CH340/N16R8 串口，没有上传固件，没有验证实体风扇、继电器、蜂鸣器、RGB 或舵机动作，也没有公网部署。
- 固定 GPIO、两室两厅一卫沙盘和低压继电器安全边界均未改变。
- 任务十完整 G1 汇总尚未执行，因此项目状态仍低于 `mock-passed`。
